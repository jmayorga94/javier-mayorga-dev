Los sistemas financieros regulados imponen restricciones que las arquitecturas event-driven comúnmente ignoran: auditoría completa, idempotencia garantizada y trazabilidad en tiempo real. Después de diseñar e implementar estas arquitecturas para una institución financiera internacional y una empresa de software financiero, tengo una perspectiva clara sobre cuándo funcionan y cuándo se convierten en deuda técnica.

## El problema de fondo

Las arquitecturas event-driven son atractivas porque desacoplan productores y consumidores. En sistemas financieros regulados, ese desacoplamiento crea un problema: **necesitas saber exactamente qué pasó, cuándo y por qué**, y ese conocimiento no puede depender de que todos los consumidores estén funcionando en el momento del evento.

<Callout type="warning">
  Un sistema bancario que perdió eventos porque el consumidor estaba caído en el momento de la
  transacción no es un sistema financiero, es un sistema con bugs regulatorios.
</Callout>

## Los tres pilares que no son negociables

### 1. Event sourcing como fuente de verdad

En lugar de guardar estado mutable, guardas una secuencia inmutable de eventos. El estado actual es siempre una proyección de esa secuencia. Esto resuelve el problema de auditoría de raíz: el log de eventos es la auditoría.

### 2. Idempotencia en todos los consumidores

Cada evento puede ser procesado más de una vez. Los consumidores deben producir el mismo resultado independientemente de cuántas veces reciban el mismo evento. Implementarlo con Azure Service Bus requiere:

- Rastrear `MessageId` en una tabla de idempotencia
- Retornar inmediatamente si el mensaje ya fue procesado
- Usar transacciones distribuidas para operaciones que afectan múltiples sistemas

### 3. Outbox pattern para consistencia garantizada

El problema clásico: guardas en base de datos y publicas en el bus. Si el sistema falla entre los dos, tienes inconsistencia. El outbox pattern resuelve esto guardando el evento en la misma transacción que el estado, y un proceso separado lo publica al bus.

<Callout type="info">
  En Azure, esto se implementa con Azure SQL + Service Bus + Azure Functions con un timer
  trigger que procesa el outbox cada pocos segundos.
</Callout>

## Lo que aprendí en una institución financiera internacional

En el proyecto de modernización de una institución financiera internacional, el sistema procesaba transacciones financieras reguladas bajo estrictos requisitos de cumplimiento normativo. Los requisitos de auditoría exigían trazabilidad completa de cualquier transacción en los últimos 10 años.

La decisión de usar event sourcing no fue una preferencia arquitectural, fue la única forma de cumplir los requisitos regulatorios sin construir un sistema de auditoría paralelo que habría costado el doble de mantener.

## Cuándo no usar event-driven

No todo sistema financiero necesita esta complejidad. Usa una arquitectura más simple cuando:

- El volumen es bajo y los SLAs son relajados
- El equipo no tiene experiencia con eventual consistency
- Los requisitos de auditoría se satisfacen con logging simple

La complejidad debe estar justificada por el problema que resuelve.
