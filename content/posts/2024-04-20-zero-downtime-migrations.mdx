"Zero-downtime migration" is on every architecture slide deck. The implementation is almost never described honestly. Here's what it actually takes.

## The two problems that are usually conflated

Zero-downtime means two different things, and you need different strategies for each:

1. **Infrastructure migration** — moving compute, storage, or network topology without service interruption
2. **Database schema migration** — changing live schemas without taking the database offline

Both are achievable. Neither is simple. Conflating them leads to migration plans that solve one and fail on the other.

## Infrastructure: the traffic split pattern

The reliable approach for infrastructure migration is a controlled traffic split with health validation at each step:

1. New infrastructure runs in parallel, receiving 0% of traffic
2. Smoke tests validate the new environment
3. Traffic shifts incrementally: 5% → 25% → 50% → 100%
4. Old infrastructure kept warm for rollback window (typically 24-72 hours)
5. Old infrastructure decommissioned after stability confirmed

<Callout type="info">
  Azure Front Door and Application Gateway both support weighted routing that makes this
  pattern straightforward to implement. The critical piece is defining the health checks
  that gate each traffic increment.
</Callout>

The part documentation skips: **the rollback plan must be tested before you migrate forward.** If you haven't validated that 100% → 0% traffic shift works cleanly, you don't have zero-downtime migration — you have zero-downtime migration with untested recovery.

## Database schema: expand/contract

The expand/contract pattern is the standard approach for zero-downtime schema changes:

**Expand phase** — make backward-compatible changes only:
- Add new columns as nullable
- Add new tables
- Add new indexes

**Migrate phase** — backfill data without blocking reads/writes

**Contract phase** — remove the old structure after the application is fully on the new schema

<Callout type="warning">
  The common failure: running the contract phase too quickly. If the old application version is
  still deployed anywhere (older containers not yet replaced, canary instances), it will break
  when the old column disappears. Wait for full deployment propagation before contracting.
</Callout>

## What nobody tells you about regulated environments

In regulated financial systems, you have a third constraint: **the migration itself must be auditable.**

This means:

- Change scripts must be version-controlled and immutable once reviewed
- Every migration step needs a timestamp, executor identity, and outcome record
- Rollback scripts must exist and be tested for every forward migration
- A migration that partially succeeds and partially fails must have a documented recovery procedure

This isn't optional. I've seen migrations fail regulatory review not because the technical approach was wrong, but because the audit trail was incomplete.

## The IDB case

The IDB legacy-to-HA migration involved a financial system processing regulated transactions with a four-hour daily maintenance window as the only allowed downtime. The requirement was to eliminate that window entirely.

The solution used traffic split at the application layer (Azure Application Gateway) and expand/contract for four schema changes that had accumulated over two years. The migration ran over six weeks in production, with zero service interruptions.

The part that took the most time wasn't the technical implementation — it was the change documentation required by the regulatory review process.
