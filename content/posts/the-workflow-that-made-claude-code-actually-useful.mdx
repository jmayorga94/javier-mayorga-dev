I built this entire site, design system, architecture diagrams, case studies, MDX pipeline, Supabase reactions, OG image generation, using Claude Code. Not as an experiment. As my primary tool.

Honestly? It didn't work. And the problem wasn't Claude. It was me.

---

## The biggest mistake: treating it like a chat window

Claude Code is not a smarter version of ChatGPT in a terminal. The moment you start treating it like a chat window, pasting code, asking questions, getting answers, you're leaving most of the value on the table.

The moment things clicked was when I stopped chasing better outputs and started thinking about how I was setting up the work.

Here's what I changed. 👇

---

## CLAUDE.md, the most important file in your repo

`CLAUDE.md` is a context file that Claude Code reads at the start of every session. Think of it as the onboarding document you'd write for a new developer joining the project, except this one gets read every single time.

Instead of jumping straight to "build me X," I started every feature with a conversation: *help me get my ideas straight and guide me to my vision.* I wasn't asking Claude to build anything yet. I was using it to explore what I actually wanted.

- **Who you are and what the project is** not generic, specific. "Personal brand site for a cloud architect, targeting senior roles and consulting engagements."
- **Tech stack decisions** not just what, but why. "Next.js 15 App Router, not Pages Router. MDX with next-mdx-remote, not contentlayer."
- **Design system** exact values. Colors, spacing, typography scale. Copy-pasted from your design source, not paraphrased.
- **Rules and constraints** "never commit directly to main", "TypeScript always, no plain JS", "follow the mockup at reference/mockup.html"
- **Component patterns** the established patterns so Claude doesn't invent new ones

When I started the design exploration first, describing what I wanted, letting Claude ask clarifying questions, narrowing down to one clear direction, everything downstream got easier. Claude had a real vision to hold in memory across every task, the design stayed consistent, and I stopped second-guessing mid-build.

<Callout type="tip">⏱️ 20 minutes of design conversation saves hours of drift.</Callout>

---

## tasks.md, keeping Claude oriented across sessions

Claude Code doesn't have memory between sessions. Every time you start a session, you're starting fresh. `tasks.md` solves this.

My workflow:

1. Before a session: review and update tasks.md with what's next
2. At session start: reference it directly, "look at tasks.md, let's work on Phase 4"
3. During the session: mark tasks as done as they complete
4. After the session: update remaining tasks and add anything discovered

The format I use is simple, phase headers, checkboxes, clear one-line descriptions. No project management overhead. The file is there to orient the agent, not to impress anyone.

<Callout type="info">📌 I treat it as a hard rule: plan first, always. A few minutes of planning saves hours of rework.</Callout>

---

## 🧠 3. Persist Context Across Sessions

Since every Claude Code session starts fresh, without a way to persist context you end up re-explaining the project from scratch every time.

**`superpowers:brainstorming`** before any feature work. This forces a design conversation before any code is written. I describe what I want to build, and the skill drives a structured exploration: clarifying questions, 2-3 approaches with trade-offs, design proposal, approval. The hard gate here is real, no code until design is approved.

- 🎨 **Design system** — exact color values, typography scale, the one accent color (`#1D9E75`) and that nothing else is allowed
- 📝 **Language rules** — "TypeScript always, no plain JS," "App Router only, never Pages Router patterns"
- 🧩 **Component rules** — "server components by default, `use client` only when required"
- 🌿 **Git workflow** — branch naming (`feature/<task>-<description>`), commit format (`feat(scope): imperative summary`), allowed scopes
- ✅ **Verification commands** — `npm run build && npm run lint` must pass before any commit
- 📂 **Reference to other docs** — `@docs/design.md`, `@docs/architecture.md` so Claude pulls those into context automatically
- 🚫 **Hard constraints** — "never add dependencies without a clear reason," "never commit directly to main"

**`superpowers:writing-plans`** once the design is approved. This produces a detailed, task-by-task implementation plan with exact file paths, complete code for each step, and commit checkpoints. The plan is saved to `docs/superpowers/plans/` and committed.

**`superpowers:verification-before-completion`** before claiming anything is done. This forces actual verification commands to run before a success claim. No "I believe this works", show the passing output.

<Callout type="tip">🧠 My threshold: any decision that took more than 30 seconds to make probably belongs in `CLAUDE.md`.</Callout>

## 🏗️ 4. Document Architectural Decisions

As you build, you make decisions. Why this component structure and not that one. Why a certain API is shaped the way it is. Why you chose one approach over something simpler.

My pattern:
- `main` is always deployable (Vercel auto-deploys on push to main)
- All work happens on `feature/<short-description>`
- Commits are small and describe intent, not just action, "feat: add comingSoon variant to BlogCard" not "update BlogCard"
- PRs use `gh pr create` with a proper description, Claude Code writes these well

`architecture.md` is where I capture that history. Not a formal document, just clear notes about what was decided and why. When a new requirement comes in weeks later, I reference it and Claude immediately has the context it needs. No re-explaining, no accidentally undoing something that was already thought through.

> Coming back to the project after a break, that file is the fastest way back into the mental model.

## ✅ 5. Write Requirements Before Coding

For tasks that are genuinely independent, building two components simultaneously, running analysis while writing code, `AGENTS.md` documents parallel agent conventions.

When I was vibe coding, I'd describe a feature and Claude would build something. Sometimes exactly right. Often close but off in small ways that required rework. I kept thinking Claude was missing details, and it was, but the real problem was I hadn't written them down clearly myself.

`requirements.md` forced me to be specific before building anything. Feature by feature, with acceptance criteria. Not because Claude needs a formal spec, but because writing it out forced me to think through edge cases I'd otherwise discover mid-build and have to explain on the fly.

A simple example looks like this:

```
## Blog reaction bar

**Goal:** Let readers react to a post with an emoji.

Acceptance criteria:
- [ ] Displays 3 reaction options (👍 🔥 💡)
- [ ] Count updates optimistically on click
- [ ] Persists to Supabase on interaction
- [ ] Works without being logged in
- [ ] Does not allow more than one reaction per user per post
```

Nothing fancy. 🎯 Just enough that both you and Claude know exactly when it's done.

Once I had them written down, Claude had real targets to work from. I could verify against actual criteria instead of gut feel, and when something was off the conversation got useful fast — "this doesn't meet criterion 3" is much cleaner than "this doesn't feel right."

<Callout type="info">💡 Writing requirements forces you to think clearly before you build, and that clarity matters more than any prompt.</Callout>

---

If you're starting to build something real with AI tools, the magic isn't in the prompts — it's in the structure you build around them. Give the tool a clear vision, a plan, and the right context and it will move fast. Skip that setup and you'll spend more time correcting than building.

I'm genuinely impressed by what Claude can create. But what excites me more is realizing that the tool is only as good as the clarity we bring to it. The ideas, the vision, the decisions ✨ — those still come from us. Claude just helps us get there faster.

1. **Write CLAUDE.md before the first prompt.** Spend 30 minutes on it. It pays back immediately.
2. **Set up tasks.md as your first file.** Structure your phases before building anything.
3. **Never skip the brainstorming step.** Even for small things. Especially for small things, those are where assumptions sneak in.
4. **Commit frequently, with clear messages.** Your future self and Claude will thank you.
5. **Add to CLAUDE.md as you go.** Every decision that took more than 30 seconds to make belongs there.

The pattern that surprised me most: the biggest productivity gains aren't from Claude writing faster code. They're from the structure the workflow imposes, the forced design step, the written plan, the small commits. Claude Code made me a more disciplined developer, not just a faster one.
